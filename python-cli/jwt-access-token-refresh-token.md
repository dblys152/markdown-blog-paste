# JWT 기반 인증에서 Access Token과 Refresh Token을 나누는 이유

JWT 기반 인증을 구현하다 보면 자연스럽게 Access Token과 Refresh Token을 나누는 구조를 접하게 됩니다.

처음에는 단순히 이렇게 생각할 수 있습니다.

> “JWT 하나만 발급해서 쓰면 되는 것 아닌가?”

실제로 로그인 후 JWT를 하나 발급하고, 클라이언트가 API 요청마다 이 토큰을 보내도록 만들면 인증 자체는 동작합니다. 서버는 토큰을 검증하고, 사용자를 식별한 뒤 요청을 처리할 수 있습니다.

하지만 실무에서는 인증이 단순히 “로그인 여부 확인”으로 끝나지 않습니다.

토큰이 탈취되었을 때 어떻게 할 것인지, 사용자가 계속 로그인 상태를 유지해야 하는지, 로그아웃이나 강제 만료를 어떻게 처리할 것인지, 보안과 사용성 사이에서 어떤 기준을 가져갈 것인지까지 함께 고민해야 합니다.

이 글에서는 JWT 기반 인증에서 Access Token과 Refresh Token을 왜 나누는지, 그리고 두 토큰을 나눌 때 어떤 설계 판단이 필요한지 정리해보겠습니다.

---

## 1. JWT 하나만 사용하면 어떤 문제가 생길까?

JWT는 기본적으로 서버가 토큰 안에 사용자 식별 정보와 만료 시간 등을 담아 서명한 토큰입니다.

서버는 요청이 들어올 때마다 토큰의 서명을 검증하고, 만료 여부를 확인한 뒤 인증된 사용자로 처리할 수 있습니다.

이 구조의 장점은 명확합니다.

서버가 매 요청마다 세션 저장소를 조회하지 않아도 됩니다. 토큰 자체에 필요한 정보가 들어 있기 때문에, 인증 처리를 비교적 단순하게 구성할 수 있습니다.

하지만 JWT 하나만 사용하면 만료 시간 설정에서 문제가 생깁니다.

토큰 만료 시간을 길게 잡으면 사용자는 편합니다. 한 번 로그인하면 오랫동안 다시 로그인하지 않아도 됩니다. 하지만 토큰이 탈취되었을 때 공격자도 그 시간 동안 계속 API를 호출할 수 있습니다.

반대로 토큰 만료 시간을 짧게 잡으면 보안상 유리합니다. 탈취되더라도 사용할 수 있는 시간이 짧기 때문입니다. 하지만 사용자는 자주 로그아웃되거나, 계속 다시 로그인해야 하는 불편을 겪게 됩니다.

결국 JWT 하나만 사용할 때의 핵심 고민은 다음입니다.

> 토큰 수명을 길게 가져가면 보안이 약해지고, 짧게 가져가면 사용성이 나빠집니다.

Access Token과 Refresh Token을 나누는 이유는 이 충돌을 완화하기 위해서입니다.

---

## 2. Access Token은 API 접근 권한을 증명하는 토큰이다

Access Token은 실제 API 요청에 사용되는 토큰입니다.

클라이언트는 API를 호출할 때 Authorization 헤더 등에 Access Token을 담아 보냅니다. 서버는 이 토큰을 검증하고, 사용자를 식별한 뒤 해당 요청을 처리합니다.

Access Token의 역할은 명확합니다.

> “이 요청을 보낸 사용자가 인증된 사용자이며, API에 접근할 수 있는가?”

따라서 Access Token은 자주 사용됩니다. 거의 모든 인증이 필요한 API 요청에 포함됩니다.

그만큼 노출 가능성도 상대적으로 높습니다. 브라우저, 모바일 앱, 네트워크 환경, 로그, 프록시, 클라이언트 저장소 등 여러 지점에서 실수로 노출될 수 있습니다.

그래서 Access Token은 보통 짧은 만료 시간을 가집니다.

예를 들어 10분, 30분, 1시간처럼 비교적 짧은 수명을 부여합니다. 이 시간은 서비스 성격과 보안 요구사항에 따라 달라질 수 있습니다.

Access Token을 짧게 가져가면 탈취되었을 때 피해 시간을 제한할 수 있습니다.

하지만 이것만으로는 사용성이 떨어집니다. Access Token이 만료될 때마다 사용자가 다시 로그인해야 한다면 일반적인 서비스에서는 불편이 큽니다.

그래서 Refresh Token이 필요해집니다.

---

## 3. Refresh Token은 로그인 상태를 연장하기 위한 토큰이다

Refresh Token은 API 요청에 직접 사용하는 토큰이 아닙니다.

Refresh Token의 주된 역할은 Access Token이 만료되었을 때 새로운 Access Token을 발급받는 것입니다.

즉, Access Token이 “API 접근 권한”을 증명한다면, Refresh Token은 “아직 로그인 상태를 연장할 수 있는가”를 판단하는 데 사용됩니다.

흐름은 대략 다음과 같습니다.

```text
1. 사용자가 로그인한다.
2. 서버가 Access Token과 Refresh Token을 발급한다.
3. 클라이언트는 API 요청 시 Access Token을 사용한다.
4. Access Token이 만료된다.
5. 클라이언트는 Refresh Token으로 새 Access Token을 요청한다.
6. 서버는 Refresh Token을 검증하고 새 Access Token을 발급한다.
```

이렇게 나누면 Access Token의 수명은 짧게 가져가면서도, 사용자는 매번 다시 로그인하지 않아도 됩니다.

여기서 중요한 점은 Refresh Token이 Access Token보다 더 민감한 토큰이라는 점입니다.

Access Token은 탈취되더라도 짧은 시간 동안만 사용할 수 있습니다. 하지만 Refresh Token이 탈취되면 공격자는 Access Token을 계속 재발급받을 수 있습니다.

따라서 Refresh Token은 단순히 “만료 시간이 긴 JWT” 정도로 보면 안 됩니다.

Refresh Token은 별도의 저장, 검증, 폐기 전략이 필요한 인증 자산에 가깝습니다.

---

## 4. 두 토큰을 나누는 핵심 이유는 책임 분리다

Access Token과 Refresh Token을 나누는 이유를 단순히 “수명이 다르기 때문”이라고만 보면 조금 부족합니다.

더 정확하게 보면 두 토큰은 책임이 다릅니다.

Access Token의 책임은 API 요청을 인증하는 것입니다. 그래서 요청마다 사용되고, 짧은 수명을 가지며, 서버는 주로 서명과 만료 시간을 검증합니다.

Refresh Token의 책임은 인증 세션을 연장하는 것입니다. 그래서 자주 사용되지 않고, 더 긴 수명을 가지며, 서버는 재발급 가능 여부를 더 신중하게 판단해야 합니다.

정리하면 다음과 같습니다.

| 구분 | Access Token | Refresh Token |
|---|---|---|
| 주된 목적 | API 접근 인증 | Access Token 재발급 |
| 사용 빈도 | 매 API 요청마다 사용 | Access Token 만료 시 사용 |
| 만료 시간 | 짧게 설정 | 상대적으로 길게 설정 |
| 탈취 시 영향 | 만료 전까지 API 접근 가능 | Access Token 재발급 가능 |
| 서버 관리 필요성 | 낮거나 선택적 | 상대적으로 높음 |

이렇게 역할을 분리하면 설계 기준도 명확해집니다.

Access Token은 빠르게 검증하고 짧게 쓰는 토큰으로 설계합니다. Refresh Token은 재발급 권한을 가진 민감한 토큰으로 보고, 저장과 폐기 전략을 함께 설계합니다.

---

## 5. Refresh Token은 서버에서 관리하는 것이 안전하다

JWT의 장점 중 하나는 서버가 상태를 저장하지 않아도 된다는 점입니다.

하지만 Refresh Token까지 완전히 무상태로 처리하면 문제가 생길 수 있습니다.

예를 들어 Refresh Token을 JWT로 발급하고, 서버가 별도로 저장하지 않는다고 가정해보겠습니다. 그러면 서버는 토큰의 서명과 만료 시간만 검증할 수 있습니다.

이 경우 사용자가 로그아웃해도 이미 발급된 Refresh Token을 즉시 무효화하기 어렵습니다. 비밀번호 변경, 계정 정지, 관리자 강제 로그아웃 같은 상황에서도 마찬가지입니다.

물론 블랙리스트 방식으로 막을 수도 있지만, 결국 서버 쪽 상태 관리가 필요해집니다.

그래서 실무에서는 Refresh Token을 서버 저장소에 보관하는 방식을 자주 사용합니다.

예를 들어 Redis나 DB에 사용자 ID, Refresh Token 식별자, 만료 시간, 발급 시각, 디바이스 정보 등을 저장할 수 있습니다.

```text
refresh_token_id
user_id
device_id
expires_at
created_at
revoked
```

이렇게 관리하면 로그아웃 시 해당 Refresh Token을 삭제하거나 폐기 상태로 바꿀 수 있습니다. 특정 디바이스에서만 로그아웃하거나, 전체 기기 로그아웃도 구현할 수 있습니다.

결국 Access Token은 상대적으로 무상태에 가깝게 운영하더라도, Refresh Token은 상태를 관리하는 쪽이 실무적으로 더 안전합니다.

---

## 6. Refresh Token 재발급에도 정책이 필요하다

Refresh Token을 사용한다고 해서 단순히 만료 시간만 길게 주면 되는 것은 아닙니다.

재발급 과정에도 정책이 필요합니다.

대표적으로 고민할 부분은 다음과 같습니다.

첫째, Refresh Token을 매번 새로 발급할 것인지입니다.

Access Token을 재발급할 때 Refresh Token도 함께 갱신하는 방식을 Refresh Token Rotation이라고 합니다. 이 방식은 기존 Refresh Token을 재사용하지 못하게 만들 수 있어 탈취 대응에 유리합니다.

예를 들어 클라이언트가 Refresh Token A로 새 Access Token을 요청하면, 서버는 Access Token B와 Refresh Token B를 새로 발급합니다. 그리고 Refresh Token A는 폐기합니다.

이후 누군가 Refresh Token A를 다시 사용하려고 하면, 서버는 토큰 재사용 공격 가능성을 의심할 수 있습니다.

둘째, 여러 기기 로그인을 허용할 것인지입니다.

사용자가 PC, 모바일, 태블릿에서 동시에 로그인할 수 있다면 Refresh Token을 사용자 단위로 하나만 관리하기 어렵습니다. 이 경우 디바이스나 세션 단위로 Refresh Token을 관리하는 편이 자연스럽습니다.

셋째, Refresh Token 만료 시간을 어떻게 가져갈 것인지입니다.

너무 짧으면 사용성이 떨어지고, 너무 길면 탈취 시 위험이 커집니다. 서비스의 보안 수준, 사용자 경험, 규제 요구사항에 따라 기준을 정해야 합니다.

넷째, 재발급 API를 어떻게 보호할 것인지입니다.

Refresh Token을 사용하는 API는 Access Token 재발급 권한을 가진 중요한 API입니다. 단순히 토큰만 검증할 것이 아니라, 필요한 경우 요청 빈도 제한, 디바이스 정보 확인, 비정상 접근 감지 같은 추가 정책도 고려할 수 있습니다.

---

## 7. Access Token에는 너무 많은 정보를 담지 않는 것이 좋다

JWT는 토큰 안에 정보를 담을 수 있습니다.

그래서 사용자 ID, 권한, 이름, 이메일, 조직 ID 등을 토큰에 넣고 싶은 유혹이 생깁니다. 하지만 Access Token은 클라이언트가 계속 들고 다니는 값입니다.

JWT는 서명된 토큰이지, 기본적으로 암호화된 토큰이 아닙니다. Base64Url로 인코딩되어 있을 뿐이므로 Payload는 누구나 디코딩해서 볼 수 있습니다.

따라서 민감 정보는 Access Token에 담지 않는 것이 좋습니다.

또한 토큰에 너무 많은 정보를 담으면 정보 변경 시점과 토큰 만료 시점 사이에 불일치가 생깁니다.

예를 들어 사용자의 권한이 변경되었는데 기존 Access Token이 아직 살아 있다면, 토큰 안의 권한 정보는 이전 상태일 수 있습니다. Access Token 만료 시간이 짧다면 이 문제의 영향이 줄어들지만, 완전히 사라지는 것은 아닙니다.

그래서 Access Token에는 보통 최소한의 식별 정보와 권한 판단에 필요한 제한적인 정보만 담는 것이 좋습니다.

```text
sub: 사용자 식별자
roles: 제한적인 권한 정보
iat: 발급 시각
exp: 만료 시각
jti: 토큰 식별자
```

중요한 권한 판단이 필요하거나 실시간성이 중요한 정보라면, 토큰 내용만 믿기보다 서버 저장소를 조회하는 방식도 함께 고려해야 합니다.

---

## 8. 로그아웃을 어떻게 볼 것인지도 설계 기준이 된다

JWT 기반 인증에서 자주 헷갈리는 지점 중 하나가 로그아웃입니다.

세션 기반 인증에서는 서버 세션을 삭제하면 로그아웃 처리가 비교적 명확합니다.

하지만 JWT는 이미 발급된 토큰이 클라이언트에 있습니다. 서버가 Access Token을 별도로 저장하지 않는 구조라면, 로그아웃 시점에 이미 발급된 Access Token을 즉시 무효화하기 어렵습니다.

그래서 JWT 기반 인증에서 로그아웃은 보통 다음처럼 설계합니다.

```text
1. 클라이언트에 저장된 Access Token, Refresh Token을 삭제한다.
2. 서버에 저장된 Refresh Token을 삭제하거나 폐기 처리한다.
3. Access Token은 짧은 만료 시간에 의해 자연스럽게 만료되도록 둔다.
```

이 방식은 완벽한 즉시 차단은 아닙니다. 로그아웃 직후에도 기존 Access Token이 만료되기 전까지는 이론적으로 사용 가능할 수 있습니다.

이 문제가 허용되지 않는 서비스라면 Access Token 블랙리스트, 토큰 버전, 사용자 세션 버전, 강제 만료 체크 같은 추가 설계가 필요합니다.

다만 모든 서비스가 이 정도의 강한 통제를 필요로 하지는 않습니다.

중요한 것은 “로그아웃을 어떤 수준으로 보장할 것인가”를 먼저 정하는 것입니다.

단순히 JWT를 쓰기 때문에 로그아웃이 어렵다고 볼 것이 아니라, Access Token의 짧은 수명과 Refresh Token의 서버 관리로 어느 정도까지 위험을 줄일 것인지 판단해야 합니다.

---

## 9. 정리: 두 토큰을 나누는 이유는 보안과 사용성의 균형이다

Access Token과 Refresh Token을 나누는 이유는 단순히 JWT 예제 코드에서 그렇게 하기 때문이 아닙니다.

핵심은 보안과 사용성의 균형입니다.

Access Token은 API 요청마다 사용되기 때문에 짧게 가져가야 합니다. 그래야 탈취되었을 때 피해 시간을 줄일 수 있습니다.

Refresh Token은 사용자가 계속 로그인 상태를 유지할 수 있도록 돕습니다. 하지만 Access Token을 재발급할 수 있는 권한을 가지기 때문에 더 신중하게 관리해야 합니다.

정리하면 다음과 같습니다.

1. JWT 하나만 사용하면 만료 시간을 길게 할지 짧게 할지의 충돌이 생깁니다.
2. Access Token은 API 접근을 인증하는 토큰이므로 짧은 수명을 가지는 것이 좋습니다.
3. Refresh Token은 Access Token을 재발급하기 위한 토큰이며, 더 민감하게 관리해야 합니다.
4. Refresh Token은 서버 저장소에 보관하고 로그아웃, 강제 만료, 디바이스별 세션 관리 기준을 함께 설계하는 편이 안전합니다.
5. Refresh Token Rotation을 적용하면 탈취나 재사용 공격을 감지하는 데 도움이 됩니다.
6. Access Token에는 민감 정보나 변경 가능성이 큰 정보를 과하게 담지 않는 것이 좋습니다.
7. JWT 기반 로그아웃은 “즉시 무효화”를 어디까지 보장할 것인지에 따라 설계가 달라집니다.
